# Code Atlas — Design Specification

> **Status:** Draft
> **Date:** 2026-07-05
> **Author:** AI-assisted design session

---

## 1. Overview

### 1.1 What Is Code Atlas?

Code Atlas is a **CLI-based code knowledge graph tool** that transforms large codebases into a queryable directed multigraph. It uses deterministic static analysis (AST parsing via tree-sitter) to extract structural entities (modules, classes, functions, variables) and their relationships (imports, calls, inheritance, type usage), then stores them in a portable SQLite-backed graph. Users query the graph via CLI commands to understand dependencies, trace workflows, assess blast radius, detect dead code, and map service boundaries.

### 1.2 Problem Statement

Developers working with large codebases (10K-50K files) face recurring challenges:
- **Understanding a new codebase:** Where does a request enter? What calls what? How are packages organized?
- **Assessing change impact:** If I modify this function, what breaks downstream?
- **Refactoring safely:** What depends on this class? Can I move it without breaking consumers?
- **Finding dead code:** What functions are never called? What imports are unused?
- **Service decomposition:** In a monorepo, where are the natural service boundaries?
- **Migration planning:** What are the transitive dependencies of this module? What would I need to extract?

These tasks require understanding the full codebase structure, not just individual files. Existing tools either require heavy infrastructure (Neo4j), depend on expensive LLM APIs, or lack the depth of analysis needed.

### 1.3 Goals

1. **Zero infrastructure** — no databases to install, no servers to run, no API keys needed
2. **Deterministic** — pure static analysis, no LLM dependency, reproducible results
3. **Fast** — handle 50K-file monorepos; incremental updates in seconds
4. **Portable** — everything stored in a `.code-atlas/` directory alongside the project
5. **Extensible** — adding a new language requires writing one Python module
6. **Comprehensive** — 8 analysis capabilities covering the most common codebase understanding tasks
7. **Local-only** — no network calls, no cloud dependencies, no git integration required

### 1.4 Non-Goals

- **IDE integration** — Code Atlas is CLI-only for v1. IDE plugins are a future consideration.
- **Real-time analysis** — Code Atlas runs on-demand (`code-atlas analyze`), not as a file watcher.
- **Semantic/LLM analysis** — No natural language queries, no AI-inferred relationships. Pure static analysis.
- **Build system integration** — Code Atlas analyzes source code, not build artifacts or dependency manifests (pom.xml, requirements.txt).
- **Cross-language call resolution** — Python calling Java (or vice versa) is not resolved in v1.
- **MCP server mode** — v1 is CLI-only. MCP protocol support is a future extension.

### 1.5 Target Users

- Developers onboarding to large codebases
- Tech leads assessing refactoring impact
- Architects mapping service boundaries in monorepos
- Anyone performing code reviews, migrations, or bug investigations in large Python/Java projects

## 2. Architecture

### 2.1 High-Level Architecture

Code Atlas follows a **3-layer pipeline** architecture:

```
┌─────────────────────────────────────────────────────┐
│                    CLI Layer                          │
│   (click-based commands: init, analyze, query,       │
│    blast-radius, visualize, report, etc.)             │
└──────────────────────┬──────────────────────────────┘
                       │
┌──────────────────────▼──────────────────────────────┐
│               Analysis Engine                        │
│   8 language-agnostic analyzers operating on the     │
│   graph topology (dependency, call graph, blast      │
│   radius, hierarchy, dead code, boundaries, flow     │
│   tracer, service mapper)                            │
└──────────────────────┬──────────────────────────────┘
                       │
┌──────────────────────▼──────────────────────────────┐
│                Graph Core                            │
│   ┌────────────────┐  ┌───────────────────────────┐ │
│   │  Graph Model   │  │ Persistence (SQLite)      │ │
│   │  (networkx     │  │ + File Hash Index         │ │
│   │   DiGraph)     │  │ + Incremental Updates     │ │
│   └───────┬────────┘  └───────────────────────────┘ │
│           │                                          │
│   ┌───────▼──────────────────────────────────────┐  │
│   │          Language Adapters                     │  │
│   │   ┌──────────────┐  ┌──────────────────┐     │  │
│   │   │   Python     │  │   Java           │     │  │
│   │   │   Adapter    │  │   Adapter        │     │  │
│   │   │ (tree-sitter │  │ (tree-sitter     │     │  │
│   │   │  -python)    │  │  -java)          │     │  │
│   │   └──────────────┘  └──────────────────┘     │  │
│   └──────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────┘
```

### 2.2 Data Flow

```
Source Files (.py, .java)
    │
    ▼
[Phase 1: Local Extraction]  ← per-file, parallelizable
    │  Tree-sitter parse → CST
    │  Adapter traverses CST → emits nodes + intra-file edges
    │
    ▼
[Phase 2: Cross-File Resolution]  ← requires full graph
    │  Resolve import targets to Module nodes
    │  Resolve function calls to Function nodes
    │  Resolve type references to Class nodes
    │  Emit CALLS, EXTENDS, IMPLEMENTS, USES_TYPE edges
    │
    ▼
[Graph Construction]
    │  networkx DiGraph populated with all nodes + edges
    │  Persisted to SQLite
    │
    ▼
[Analysis / Query]
    │  CLI commands invoke analyzers
    │  Analyzers traverse graph topology
    │  Results formatted and displayed
```

### 2.3 Key Design Decisions

| Decision | Rationale |
|---|---|
| **networkx for graph** | Rich algorithm library (BFS, DFS, shortest path, community detection), well-tested, Pythonic API. Memory is acceptable at 50K files (~2-4GB). |
| **SQLite for persistence** | Zero-config, portable, single-file, ACID, excellent Python stdlib support. WAL mode for performance. |
| **tree-sitter for parsing** | Language-agnostic, fast, produces concrete syntax trees, handles syntax errors gracefully, wide language support (30+). |
| **Protocol for adapters** | Python's `typing.Protocol` enables structural subtyping — adapters don't need to inherit from a base class, just implement the right methods. |
| **Two-phase extraction** | Phase 1 (per-file) is parallelizable. Phase 2 (cross-file) needs the full graph but is typically faster since it's just edge resolution. |
| **File-hash incremental** | SHA-256 content hashing is fast and deterministic. Only re-parse files whose content actually changed. |

## 3. Graph Data Model

### 3.1 Node Types

All node types share a common set of base properties plus type-specific properties.

#### Common Node Properties

| Property | Type | Description |
|---|---|---|
| `id` | `str` | SHA-256 hash of `{language}:{fqn}`. Deterministic, stable across runs. |
| `kind` | `str` | Node type: `Module`, `Package`, `Class`, `Function`, `Variable`, `Enum`, `Annotation` |
| `fqn` | `str` | Fully qualified name (unique within the graph) |
| `name` | `str` | Short name (e.g., `UserService`, `process_order`) |
| `language` | `str` | `"python"` or `"java"` |
| `file_path` | `str` | Absolute path to the source file |
| `line_start` | `int` | Start line in source (1-indexed) |
| `line_end` | `int` | End line in source (1-indexed) |

#### Type-Specific Properties

**Module** (a source file):
| Property | Type | Description |
|---|---|---|
| `package` | `str` | Package this module belongs to |
| `content_hash` | `str` | SHA-256 of file content |
| `loc` | `int` | Lines of code |

**Package** (a directory/package):
| Property | Type | Description |
|---|---|---|
| `path` | `str` | Directory path |
| `depth` | `int` | Nesting depth from project root |

**Class** (class, interface, abstract class):
| Property | Type | Description |
|---|---|---|
| `is_abstract` | `bool` | Whether class is abstract |
| `is_interface` | `bool` | Whether it's an interface (Java) or Protocol (Python) |
| `visibility` | `str` | `"public"`, `"private"`, `"protected"`, `"package"`, `"module"` |
| `decorators` | `list[str]` | Applied decorators/annotations |
| `superclasses` | `list[str]` | Direct superclass FQNs (for quick lookup) |
| `interfaces` | `list[str]` | Implemented interface FQNs (for quick lookup) |

**Function** (function, method, constructor):
| Property | Type | Description |
|---|---|---|
| `class_fqn` | `str \| None` | FQN of containing class (None for module-level functions) |
| `is_static` | `bool` | Whether method is static |
| `is_constructor` | `bool` | `__init__` (Python), constructor (Java) |
| `is_abstract` | `bool` | Abstract method |
| `visibility` | `str` | Access modifier |
| `parameters` | `list[dict]` | `[{"name": "x", "type": "int", "default": null}]` |
| `return_type` | `str \| None` | Return type annotation if present |
| `decorators` | `list[str]` | Applied decorators/annotations |
| `complexity` | `int` | Cyclomatic complexity (number of decision points) |
| `is_property` | `bool` | Python `@property` methods |

**Variable** (module-level or class-level variables/constants):
| Property | Type | Description |
|---|---|---|
| `type_annotation` | `str \| None` | Type hint if present |
| `is_constant` | `bool` | ALL_CAPS naming convention or `final` keyword |
| `scope` | `str` | `"module"`, `"class"`, `"instance"` |
| `value_hint` | `str \| None` | String representation of simple literal values |

**Enum**:
| Property | Type | Description |
|---|---|---|
| `values` | `list[str]` | Enum member names |
| `visibility` | `str` | Access modifier |

**Annotation** (Java annotations, Python decorators):
| Property | Type | Description |
|---|---|---|
| `arguments` | `dict` | Annotation arguments as key-value pairs |
| `target_kind` | `str` | What it decorates: `"class"`, `"method"`, `"field"` |

### 3.2 Edge Types

| Edge Type | Source Node → Target Node | Description | Confidence |
|---|---|---|---|
| `IMPORTS` | Module → Module | File imports another file/module | EXTRACTED |
| `CONTAINS` | Module → Class/Function/Variable/Enum | File contains a top-level definition | EXTRACTED |
| `DEFINES_METHOD` | Class → Function | Class defines a method | EXTRACTED |
| `DEFINES_FIELD` | Class → Variable | Class defines a field/attribute | EXTRACTED |
| `CALLS` | Function → Function | Function calls another function | EXTRACTED or INFERRED |
| `INSTANTIATES` | Function → Class | Function creates an instance of a class | EXTRACTED |
| `EXTENDS` | Class → Class | Class inheritance (single or multiple) | EXTRACTED |
| `IMPLEMENTS` | Class → Class | Interface/Protocol implementation | EXTRACTED |
| `USES_TYPE` | Function → Class | Function references a type (param, return, local var) | EXTRACTED |
| `OVERRIDES` | Function → Function | Method overrides a parent class method | INFERRED |
| `DECORATES` | Annotation → Function/Class | Decorator/annotation applied to an entity | EXTRACTED |
| `BELONGS_TO` | Module → Package | File belongs to a package | EXTRACTED |
| `DEPENDS_ON` | Package → Package | Package-level dependency (derived from IMPORTS) | INFERRED |

#### Edge Properties

| Property | Type | Description |
|---|---|---|
| `kind` | `str` | Edge type string |
| `source_id` | `str` | Source node ID |
| `target_id` | `str` | Target node ID |
| `file_path` | `str` | File where the relationship was detected |
| `line` | `int` | Line number where the relationship occurs |
| `confidence` | `str` | `"EXTRACTED"` (from AST) or `"INFERRED"` (heuristic) |

### 3.3 Fully Qualified Name (FQN) Convention

Every node is uniquely identified by a fully qualified name:

**Python:**
```
project_root.package.subpackage.module.ClassName.method_name
```
Examples:
- Module: `myapp.services.user_service`
- Class: `myapp.services.user_service.UserService`
- Method: `myapp.services.user_service.UserService.create_user`
- Function: `myapp.utils.helpers.format_date`
- Variable: `myapp.config.settings.DATABASE_URL`

**Java:**
```
com.company.package.ClassName.methodName
```
Examples:
- Module: `com.company.services.UserService` (Java file = class)
- Class: `com.company.services.UserService`
- Method: `com.company.services.UserService.createUser`
- Inner class: `com.company.services.UserService.Builder`
- Enum: `com.company.models.Status`

**FQN Resolution Rules:**
1. Python module FQN is derived from the file path relative to the project root, replacing `/` with `.` and stripping `.py`
2. Java FQN uses the `package` declaration from the source file
3. For Python files in directories without `__init__.py`, use the directory path as a namespace (not a package)
4. Node IDs are `SHA-256(language + ":" + fqn)` — deterministic and collision-resistant

## 4. Language Adapter System

### 4.1 Protocol Definition

```python
from typing import Protocol, Iterator, runtime_checkable
from code_atlas.core.models import ParsedFile, GraphNode, GraphEdge
from code_atlas.core.graph import CodeGraph

@runtime_checkable
class LanguageAnalyzer(Protocol):
    """Protocol that all language adapters must implement.
    
    Adding a new language to Code Atlas requires implementing this protocol
    in a new module under code_atlas/languages/.
    """
    
    @property
    def language(self) -> str:
        """Language identifier, e.g., 'python', 'java'."""
        ...
    
    @property
    def file_extensions(self) -> tuple[str, ...]:
        """File extensions this adapter handles, e.g., ('.py',), ('.java',)."""
        ...
    
    def can_handle(self, file_path: str) -> bool:
        """Return True if this adapter can parse the given file."""
        ...
    
    def parse_file(self, file_path: str, content: bytes) -> ParsedFile:
        """Parse a source file into a ParsedFile structure.
        
        Uses tree-sitter to produce a concrete syntax tree (CST).
        The ParsedFile contains the CST root node plus file metadata.
        Must handle syntax errors gracefully (partial parse is acceptable).
        """
        ...
    
    def extract_nodes(self, parsed: ParsedFile) -> Iterator[GraphNode]:
        """Extract graph nodes from a parsed file.
        
        Yields: Module node, Class nodes, Function nodes, Variable nodes,
        Enum nodes, Annotation nodes found in this file.
        Each node has its properties fully populated.
        """
        ...
    
    def extract_edges(self, parsed: ParsedFile) -> Iterator[GraphEdge]:
        """Extract intra-file edges from a parsed file.
        
        Yields: CONTAINS, DEFINES_METHOD, DEFINES_FIELD, DECORATES edges,
        plus IMPORTS edges with potentially unresolved targets.
        Cross-file edges (CALLS, EXTENDS, etc.) are resolved in Phase 2.
        """
        ...
    
    def resolve_references(
        self, parsed: ParsedFile, graph: 'CodeGraph'
    ) -> Iterator[GraphEdge]:
        """Resolve cross-file references using the full graph.
        
        Called during Phase 2 after all files have been parsed.
        Uses the graph to resolve import targets, function call targets,
        type references, inheritance chains.
        
        Yields: CALLS, INSTANTIATES, EXTENDS, IMPLEMENTS, USES_TYPE,
        OVERRIDES edges with resolved target node IDs.
        """
        ...
    
    def detect_entry_points(self, parsed: ParsedFile) -> Iterator[str]:
        """Identify entry point function FQNs in this file.
        
        Used by dead code detection to determine reachability roots.
        Examples: main(), __main__ block, @app.route handlers,
        @Controller methods, test methods.
        
        Yields: FQNs of entry point functions.
        """
        ...
```

### 4.2 Core Data Classes

```python
from dataclasses import dataclass, field
from typing import Any

@dataclass(frozen=True)
class GraphNode:
    """A node in the code knowledge graph."""
    id: str                           # SHA-256 of language:fqn
    kind: str                         # Module, Class, Function, Variable, Enum, Annotation
    fqn: str                          # Fully qualified name
    name: str                         # Short name
    language: str                     # "python" or "java"
    file_path: str                    # Absolute file path
    line_start: int                   # Start line (1-indexed)
    line_end: int                     # End line (1-indexed)
    properties: dict[str, Any] = field(default_factory=dict)  # Type-specific properties

@dataclass(frozen=True)
class GraphEdge:
    """An edge in the code knowledge graph."""
    kind: str                         # IMPORTS, CALLS, EXTENDS, etc.
    source_id: str                    # Source node ID
    target_id: str                    # Target node ID
    file_path: str                    # Where the relationship was detected
    line: int = 0                     # Line number
    confidence: str = "EXTRACTED"     # EXTRACTED or INFERRED
    properties: dict[str, Any] = field(default_factory=dict)

@dataclass
class ParsedFile:
    """Result of parsing a single source file."""
    file_path: str                    # Absolute path
    language: str                     # "python" or "java"
    content_hash: str                 # SHA-256 of file content
    tree: Any                         # tree-sitter Tree object
    source: bytes                     # Raw file content
    errors: list[str] = field(default_factory=list)  # Parse errors/warnings
```

### 4.3 Two-Phase Extraction Pipeline

**Phase 1 — Local Extraction (per-file, parallelizable):**

```
For each source file:
  1. Read file content, compute SHA-256 hash
  2. Check file_index: if hash unchanged, skip (incremental)
  3. Select appropriate LanguageAnalyzer based on file extension
  4. analyzer.parse_file(path, content) → ParsedFile
  5. analyzer.extract_nodes(parsed) → yield GraphNode objects
  6. analyzer.extract_edges(parsed) → yield GraphEdge objects (intra-file)
  7. analyzer.detect_entry_points(parsed) → yield entry point FQNs
  8. Collect all nodes and edges
```

Phase 1 is parallelized using `multiprocessing.Pool`. Each file is independent — no shared state between workers. Workers return serialized node/edge lists.

**Phase 2 — Cross-File Resolution (sequential, requires full graph):**

```
For each parsed file:
  1. analyzer.resolve_references(parsed, graph) → yield GraphEdge objects
  2. For IMPORTS edges: resolve target module FQN → target Module node ID
  3. For CALLS edges: resolve function name → target Function node ID
     - Use import scope + class hierarchy for resolution
  4. For EXTENDS/IMPLEMENTS edges: resolve class name → target Class node ID
  5. For USES_TYPE edges: resolve type name → target Class node ID
  6. For OVERRIDES edges: match method signatures against parent class methods
  7. Add resolved edges to graph
  8. Mark unresolvable references with confidence="INFERRED", resolved=false
```

### 4.4 Python Adapter Specifics

**Tree-sitter grammar:** `tree-sitter-python`

**Constructs handled:**

| Construct | Node Type | Notes |
|---|---|---|
| `class Foo:` | Class | Including `dataclass`, `NamedTuple`, `Protocol` |
| `class Foo(Base):` | Class + EXTENDS edge | Multiple inheritance: multiple EXTENDS edges |
| `class Foo(Protocol):` | Class (`is_interface=True`) | |
| `def foo():` | Function | Module-level function |
| `def foo(self):` | Function (`class_fqn` set) | Instance method |
| `@staticmethod` | Function (`is_static=True`) | |
| `@classmethod` | Function (decorator tracked) | |
| `@property` | Function (`is_property=True`) | |
| `@decorator` | DECORATES edge | |
| `import x` | IMPORTS edge | Target: module `x` |
| `from x import y` | IMPORTS edge + specific entity | |
| `from x import *` | IMPORTS edge (`is_wildcard=True`) | |
| `from .module import x` | IMPORTS edge | Relative import resolved |
| `x = value` | Variable (module-level) | |
| `self.x = value` | Variable (instance field) | via DEFINES_FIELD |
| `x: int = value` | Variable with type annotation | |
| `X = Enum(...)` | Enum | |
| `foo()` | CALLS edge | Resolved in Phase 2 |
| `Foo()` | INSTANTIATES edge | Resolved in Phase 2 |
| `x: Foo` | USES_TYPE edge | Resolved in Phase 2 |

**Entry point detection:**
- Functions named `main`
- `if __name__ == "__main__":` blocks
- Functions decorated with `@app.route`, `@click.command`, `@pytest.fixture`, `@pytest.mark.*`, `def test_*`
- Functions decorated with `@celery.task`, `@dramatiq.actor`

**Python-specific resolution:**
- Relative imports resolved using `__init__.py` package structure
- `__all__` used to determine public API for wildcard imports
- Method Resolution Order (MRO) used for `OVERRIDES` detection

### 4.5 Java Adapter Specifics

**Tree-sitter grammar:** `tree-sitter-java`

**Constructs handled:**

| Construct | Node Type | Notes |
|---|---|---|
| `class Foo {}` | Class | |
| `abstract class Foo {}` | Class (`is_abstract=True`) | |
| `interface Foo {}` | Class (`is_interface=True`) | |
| `enum Foo {}` | Enum | |
| `record Foo() {}` | Class (treated as class) | |
| `class Foo extends Bar` | Class + EXTENDS edge | |
| `class Foo implements I` | Class + IMPLEMENTS edge | Multiple interfaces |
| `void method() {}` | Function | |
| `Foo() {}` | Function (`is_constructor=True`) | |
| `static void m() {}` | Function (`is_static=True`) | |
| `@Override` | OVERRIDES edge | |
| `@Annotation` | DECORATES edge | |
| `import x.y.z;` | IMPORTS edge | |
| `import static x.y.z;` | IMPORTS edge (static import) | |
| `import x.y.*;` | IMPORTS edge (`is_wildcard=True`) | |
| `private int x;` | Variable | With visibility |
| `static final X = v;` | Variable (`is_constant=True`) | |
| `class Inner {}` | Class (nested) | FQN: `Outer.Inner` |
| `new Foo()` | INSTANTIATES edge | |
| `foo()` | CALLS edge | Resolved in Phase 2 |
| `Foo x;` | USES_TYPE edge | Resolved in Phase 2 |

**Entry point detection:**
- `public static void main(String[] args)` methods
- Classes/methods annotated with `@Controller`, `@RestController`, `@Service`, `@Component`, `@Bean`
- Methods annotated with `@RequestMapping`, `@GetMapping`, `@PostMapping`, etc.
- Methods annotated with `@Test`, `@BeforeEach`, `@AfterEach` (JUnit)
- Classes annotated with `@SpringBootApplication`

**Java-specific resolution:**
- Package declarations determine FQN prefix
- Wildcard imports resolved against known classes in the graph
- Static imports resolved to specific methods/fields
- Inner class references resolved via enclosing class scope
- Generic type parameters extracted for `USES_TYPE` edges (e.g., `List<User>` → USES_TYPE to `User`)

### 4.6 Adding a New Language

To add TypeScript support (as an example):

1. **Create** `src/code_atlas/languages/typescript_analyzer.py`
2. **Implement** `TypeScriptAnalyzer` class satisfying the `LanguageAnalyzer` protocol:
   - Set `language = "typescript"`, `file_extensions = (".ts", ".tsx")`
   - Implement `parse_file()` using `tree-sitter-typescript`
   - Implement `extract_nodes()` to handle TS-specific constructs (interfaces, type aliases, enums)
   - Implement `extract_edges()` for intra-file relationships
   - Implement `resolve_references()` for cross-file resolution
   - Implement `detect_entry_points()` for entry point detection
3. **Register** in `src/code_atlas/languages/__init__.py`:
   ```python
   LANGUAGE_REGISTRY: dict[str, type[LanguageAnalyzer]] = {
       "python": PythonAnalyzer,
       "java": JavaAnalyzer,
       "typescript": TypeScriptAnalyzer,  # add this line
   }
   ```
4. **Add dependency:** `pip install tree-sitter-typescript`

No changes needed in the graph model, analysis engine, CLI, persistence layer, or visualization.

## 5. Analysis Engine

All 8 analyzers operate on the graph topology and are completely language-agnostic. They are implemented as classes inheriting from a `BaseAnalyzer` abstract base class.

### 5.1 Base Analyzer

```python
from abc import ABC, abstractmethod
from dataclasses import dataclass
from typing import Any

@dataclass
class AnalysisResult:
    """Structured result from any analyzer."""
    analyzer: str              # Analyzer name
    query: dict[str, Any]      # Input parameters
    nodes: list[dict]          # Result nodes with properties
    edges: list[dict]          # Result edges (if applicable)
    metrics: dict[str, Any]    # Computed metrics
    warnings: list[str]        # Any warnings produced

class BaseAnalyzer(ABC):
    """Base class for all analyzers."""
    
    def __init__(self, graph: CodeGraph):
        self.graph = graph
    
    @abstractmethod
    def analyze(self, **kwargs) -> AnalysisResult:
        """Run the analysis with the given parameters."""
        ...
```

### 5.2 Dependency Analyzer

**Purpose:** Answer "what does X depend on?" and "what depends on X?"

**Input:** Any entity (module, class, function) identified by FQN or file path.

**Capabilities:**
- **Forward dependencies:** What does this entity depend on? Traverse outgoing `IMPORTS`, `CALLS`, `USES_TYPE`, `EXTENDS`, `IMPLEMENTS` edges.
- **Reverse dependencies:** What depends on this entity? Traverse incoming edges of the same types.
- **Transitive closure:** Follow dependency chains to configurable depth (default: unlimited).
- **Dependency tree:** Render as an indented tree showing the full dependency hierarchy.
- **Scope filtering:** Filter results by package, language, or node type.

**Algorithm:**
1. Locate the target node by FQN or file path
2. BFS/DFS from the target following relevant edge types
3. Track depth for each discovered node
4. Apply depth limit if specified
5. Build result tree preserving parent-child relationships

**CLI:**
```
code-atlas deps <target> [--depth N] [--reverse] [--transitive] [--types module,class,function] [--format table|json|tree]
```

**Example output:**
```
Dependencies of myapp.services.UserService (depth=2):
├── myapp.models.user.User (IMPORTS)
│   ├── myapp.models.base.BaseModel (EXTENDS)
│   └── myapp.db.session.Session (IMPORTS)
├── myapp.repositories.user_repo.UserRepository (IMPORTS)
│   └── myapp.db.session.Session (IMPORTS)
└── myapp.utils.validators.validate_email (IMPORTS)

3 direct dependencies, 5 transitive (depth 2)
```

### 5.3 Call Graph Analyzer

**Purpose:** Answer "who calls whom?" — trace execution paths through the codebase.

**Input:** A function FQN.

**Capabilities:**
- **Callees:** What functions does this function call? (forward call tree)
- **Callers:** What functions call this function? (reverse call tree)
- **Full call tree:** Both directions with depth limit.
- **Call chain:** Find all call paths from function A to function B.
- **Entry point discovery:** What entry points can reach this function?

**Algorithm:**
1. Locate the target Function node
2. Traverse `CALLS` edges (forward or reverse depending on mode)
3. Track full paths for chain discovery
4. For entry point discovery: reverse BFS to find nodes with no incoming `CALLS` edges (or flagged as entry points)

**CLI:**
```
code-atlas callgraph <function-fqn> [--callers] [--callees] [--depth N] [--chain-to <target-fqn>] [--format table|json|tree]
```

**Example output:**
```

├── myapp.repositories.user_repo.UserRepository.save
│   ├── myapp.db.session.Session.add
│   └── myapp.db.session.Session.commit
├── myapp.utils.validators.validate_email
│   └── re.match (external)
└── myapp.events.publisher.publish_event
    └── myapp.queue.redis_queue.enqueue

```

### 5.4 Blast Radius Analyzer

**Purpose:** Answer "if I change X, what could break?"

**Input:** A file path, function FQN, or class FQN.

**Capabilities:**
- **Impact assessment:** All transitively affected entities, grouped by hop distance.
- **Impact levels:** Direct (1 hop), Indirect (2+ hops), with distance shown.
- **Affected files:** Unique list of files that contain affected entities.
- **Affected tests:** Filter to test files/functions in the blast radius.
- **Risk scoring:** Higher connectivity = higher risk score.

**Algorithm:**
1. Identify all nodes in the target (if target is a file, get all nodes in that file)
2. Multi-source BFS on **reverse** dependency edges: traverse incoming `CALLS`, `IMPORTS`, `USES_TYPE`, `EXTENDS`, `IMPLEMENTS` edges
3. Track distance from source for each discovered node
4. Group results by distance
5. Compute risk score: `score = Σ (1/distance) * node_connectivity`

**CLI:**
```
code-atlas blast-radius <target> [--depth N] [--tests-only] [--files-only] [--format table|json|html]
```

**Example output:**
```
Blast radius of myapp.models.user.User:

Distance 1 (direct dependents):
  myapp.services.user_service.UserService         class     EXTENDS
  myapp.repositories.user_repo.UserRepository      class     USES_TYPE
  myapp.serializers.user_serializer.UserSerializer  class     USES_TYPE

Distance 2:
  myapp.api.views.user_views.create_user           function  CALLS → UserService
  myapp.api.views.user_views.list_users            function  CALLS → UserRepository

Affected files: 4
Affected tests: tests/test_user_service.py, tests/test_user_views.py
Risk score: 7.3 (HIGH)
```

### 5.5 Type Hierarchy Analyzer

**Purpose:** Answer "what is the class hierarchy?" — inheritance and implementation trees.

**Input:** A class FQN or "all" for the full hierarchy.

**Capabilities:**
- **Ancestors:** Traverse `EXTENDS` and `IMPLEMENTS` edges upward.
- **Descendants:** Traverse `EXTENDS` and `IMPLEMENTS` edges downward (reverse).
- **Full hierarchy:** Complete tree from root classes down.
- **Overrides map:** For a class, show which methods override parent methods.
- **Diamond detection:** Find classes with multiple inheritance paths to the same ancestor.

**Algorithm:**
1. Locate the target Class node
2. For ancestors: BFS on outgoing `EXTENDS` / `IMPLEMENTS` edges
3. For descendants: BFS on incoming `EXTENDS` / `IMPLEMENTS` edges
4. For overrides: match `OVERRIDES` edges from the class's methods
5. For diamond detection: find nodes reachable via multiple paths in the ancestor tree

**CLI:**
```
code-atlas hierarchy <class-fqn> [--ancestors] [--descendants] [--all] [--overrides] [--format table|json|tree]
```

**Example output:**

### 5.6 Dead Code Detector

**Purpose:** Find unreachable code — functions never called, imports never used, orphan classes.

**Input:** Entire graph, optionally scoped to a package.

**Capabilities:**
- **Unreachable functions:** Functions with no incoming `CALLS` edges and not identified as entry points.
- **Unused imports:** Import edges where the target module/entity is never referenced beyond the import statement itself.
- **Orphan classes:** Classes with no incoming `EXTENDS`, `IMPLEMENTS`, `INSTANTIATES`, or `USES_TYPE` edges.
- **Scope filtering:** Analyze only a specific package.
- **Test awareness:** Option to include or exclude test files from analysis.

**Algorithm:**
1. **Entry point collection:** Gather all nodes flagged as entry points by language adapters.
2. **Reachability analysis:** BFS from all entry points following `CALLS` edges. Mark all reached Function nodes.
3. **Unreachable detection:** All Function nodes not reached = potentially dead.
4. **Import analysis:** For each IMPORTS edge, check if the target module/entity has any other incoming edges from the same source module (beyond the IMPORTS edge itself).
5. **Orphan analysis:** Classes with zero incoming `EXTENDS`, `IMPLEMENTS`, `INSTANTIATES`, `USES_TYPE` edges (excluding self-references).
6. **False positive reduction:**
   - Exclude `__init__`, `__str__`, `__repr__` and other dunder methods
   - Exclude methods decorated with known framework decorators
   - Exclude public methods in public classes (may be used externally)
   - Report confidence level for each finding

**CLI:**
```
code-atlas dead-code [--scope <package>] [--include-tests] [--category functions|imports|classes|all] [--format table|json]
```

**Example output:**

### 5.7 Cross-Module Boundary Analyzer

**Purpose:** Understand package structure, coupling metrics, and boundary violations.

**Input:** Entire graph, optionally focused on a specific package.

**Capabilities:**
- **Coupling matrix:** Package-to-package dependency counts.
- **Coupling metrics:** Afferent coupling (Ca), efferent coupling (Ce), instability (I = Ce / (Ca + Ce)).
- **Boundary violations:** Internal (non-public) entities accessed from outside their package.
- **Dependency direction:** Identify packages with bidirectional dependencies (potential architectural issues).
- **Package health report:** Highlight packages with high coupling, low cohesion.

**Algorithm:**
1. **Group nodes by package:** Aggregate all nodes into their containing packages.
2. **Build package graph:** Create a derived graph where packages are nodes and inter-package edges are weighted by the number of cross-package entity references.
3. **Coupling metrics:**
   - Ca (afferent) = number of packages that depend on this package
   - Ce (efferent) = number of packages this package depends on
   - I (instability) = Ce / (Ca + Ce). I=0 is maximally stable, I=1 is maximally unstable.
4. **Boundary violations:** For each cross-package edge, check if the target entity's visibility allows external access:
   - Python: entities starting with `_` are internal by convention
   - Java: `private` and `package-private` entities should not be accessed externally
5. **Bidirectional detection:** Find package pairs where edges exist in both directions.

**CLI:**
```
code-atlas boundaries [--package <pkg>] [--violations-only] [--bidirectional] [--format table|json|html]
```

**Example output:**
```

Coupling Metrics:
  Package                    Ca    Ce    Instability
  myapp.models               8     2     0.20 (stable)
  myapp.services             3     5     0.63
  myapp.api                  1     4     0.80 (unstable)
  myapp.utils                6     1     0.14 (stable)
  myapp.repositories         4     3     0.43

Boundary Violations (3):
  myapp.api.views → myapp.services._internal_helper    PRIVATE ACCESS
  myapp.services → myapp.models.user._validate_email   PRIVATE ACCESS
  myapp.api → myapp.repositories._build_query          PRIVATE ACCESS

Bidirectional Dependencies (1):
  myapp.services ↔ myapp.repositories    (potential circular architecture)
```

### 5.8 Workflow/Flow Tracer

**Purpose:** Trace a request or data flow from an entry point through the entire system.

**Input:** A starting function (entry point), optionally a target function.

**Capabilities:**
- **Forward trace:** From entry point, show all reachable functions (call tree).
- **Targeted trace:** Find all call paths from entry point to a specific target function.
- **Path enumeration:** List all distinct execution paths (with depth limit to prevent explosion).
- **Layer crossing:** Show when the flow crosses package/module boundaries.

**Algorithm:**
1. Locate the entry Function node.
2. **Forward trace:** BFS on `CALLS` edges with depth limit. Build a tree of all reachable functions.
3. **Targeted trace:** DFS with path tracking. Find all simple paths from entry to target (no cycles). Apply depth limit.
4. For each path, annotate package crossings.
5. Limit to `max_paths` (default: 20) to prevent combinatorial explosion.

**CLI:**
```
code-atlas trace <entry-function> [--to <target-function>] [--depth N] [--max-paths N] [--format table|json|tree]
```

**Example output:**
```

  → myapp.repositories.user_repo.save        [repositories]

  myapp.api.views.create_user_endpoint       [api]
```

### 5.9 Service Boundary Mapper

**Purpose:** In monorepos, identify logical service boundaries — clusters of packages that form cohesive services.

**Input:** Entire graph.

**Capabilities:**
- **Community detection:** Use Leiden algorithm to find clusters of highly-cohesive packages.
- **Service map:** Name each service boundary, list its constituent packages.
- **Inter-service interfaces:** Identify edges crossing service boundaries — these are the "API surface" between services.
- **Coupling strength:** Quantify how tightly services are coupled.
- **Decomposition suggestions:** Based on coupling metrics, suggest which services could be cleanly extracted.

**Algorithm:**
1. Build the package-level dependency graph (from the Boundary Analyzer's derived graph).
2. Apply Leiden community detection (via networkx's `community` module or `leidenalg`).
3. Each community = one logical service.
4. Name services based on dominant package names.
5. Compute inter-community edges and coupling strength.
6. Calculate extraction difficulty: `difficulty = inter_service_edges / total_edges_involving_service`.

**CLI:**
```
code-atlas services [--min-size N] [--algorithm leiden|louvain] [--format table|json|html]
```

**Example output:**

### 7.3 SQLite Schema

```sql
-- Core tables
CREATE TABLE nodes (
    id TEXT PRIMARY KEY,
    kind TEXT NOT NULL,
    fqn TEXT NOT NULL,
    name TEXT NOT NULL,
    language TEXT NOT NULL,
    file_path TEXT NOT NULL,
    line_start INTEGER DEFAULT 0,
    line_end INTEGER DEFAULT 0,
    properties TEXT NOT NULL DEFAULT '{}'   -- JSON blob
);

CREATE TABLE edges (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    kind TEXT NOT NULL,
    source_id TEXT NOT NULL,
    target_id TEXT NOT NULL,
    file_path TEXT DEFAULT '',
    line INTEGER DEFAULT 0,
    confidence TEXT NOT NULL DEFAULT 'EXTRACTED',
    properties TEXT NOT NULL DEFAULT '{}',
    FOREIGN KEY (source_id) REFERENCES nodes(id) ON DELETE CASCADE,
    FOREIGN KEY (target_id) REFERENCES nodes(id) ON DELETE CASCADE,
    UNIQUE(kind, source_id, target_id)
);

CREATE TABLE file_index (
    path TEXT PRIMARY KEY,
    content_hash TEXT NOT NULL,
    last_analyzed TEXT NOT NULL,
    language TEXT NOT NULL,
    node_count INTEGER DEFAULT 0,
    edge_count INTEGER DEFAULT 0,
    parse_errors TEXT DEFAULT '[]'         -- JSON array of error strings
);

CREATE TABLE metadata (
    key TEXT PRIMARY KEY,
    value TEXT NOT NULL
);
-- Stores: schema_version, last_full_analyze, project_root

-- Performance indexes
CREATE INDEX idx_nodes_kind ON nodes(kind);
CREATE INDEX idx_nodes_fqn ON nodes(fqn);
CREATE INDEX idx_nodes_name ON nodes(name);
CREATE INDEX idx_nodes_file ON nodes(file_path);
CREATE INDEX idx_nodes_language ON nodes(language);
CREATE INDEX idx_edges_kind ON edges(kind);
CREATE INDEX idx_edges_source ON edges(source_id);
CREATE INDEX idx_edges_target ON edges(target_id);
CREATE INDEX idx_edges_source_kind ON edges(source_id, kind);
CREATE INDEX idx_edges_target_kind ON edges(target_id, kind);
```

### 7.4 Incremental Update Strategy

```
1. Scan project directory for all source files matching configured extensions
2. For each file:
   a. Compute SHA-256 content hash
   b. Look up file in file_index table
   c. If hash matches → skip (unchanged)
   d. If hash differs → mark for re-parse (changed)
   e. If not in index → mark for parse (new)
3. For files in index but not on disk → mark for deletion (removed)
4. Delete all nodes and edges associated with changed/removed files
5. Parse changed/new files (Phase 1, parallel)
6. Insert new nodes and edges
7. Run cross-file resolution (Phase 2) for changed files
8. Update file_index with new hashes
9. Report: N files parsed, M files skipped, K files removed
```

**Performance notes:**
- File hash computation is fast (SHA-256 of file content)
- SQLite deletions are batched in transactions of 5000 rows
- Insertions use `executemany` for batch efficiency
- WAL (Write-Ahead Logging) mode enabled for concurrent reads during writes
- `PRAGMA journal_mode=WAL; PRAGMA synchronous=NORMAL;` for write performance

## 9. Performance Strategy

### 9.1 Performance Targets

| Operation | Target (50K files) |
|---|---|
| Full initial scan | 3-8 minutes |
| Incremental update (100 changed files) | 5-15 seconds |
| Query (blast radius, depth 5) | <1 second |
| Query (dead code detection) | <5 seconds |
| Graph load from SQLite to memory | 10-30 seconds |
| Visualization generation | <10 seconds |

### 9.2 Performance Techniques

| Technique | Applied To | Implementation |
|---|---|---|
| **Parallel parsing** | Phase 1 (file parsing) | `multiprocessing.Pool(workers)`. Each file is independent. Workers return serialized node/edge lists. |
| **Batch SQLite writes** | Persistence layer | `executemany()` with transaction batches of 5000 rows. |
| **WAL mode** | SQLite | `PRAGMA journal_mode=WAL` for concurrent read during write. |
| **Lazy graph loading** | CLI queries | For most queries, load only the relevant subgraph from SQLite rather than the full graph. |
| **Generator-based extraction** | Language adapters | Adapters yield nodes/edges via `Iterator`, not building full lists in memory. |
| **Index-first queries** | SQLite | Composite indexes on `(source_id, kind)` and `(target_id, kind)` for fast edge lookups. |
| **File-hash diffing** | Incremental updates | SHA-256 hash comparison — only re-parse changed files. |
| **Memoized resolutions** | Phase 2 | Cache resolved FQN → node ID mappings during reference resolution. |

### 9.3 Memory Management

At 50K files, the expected graph size:
- **Nodes:** ~200K-500K (modules + classes + functions + variables)
- **Edges:** ~500K-2M
- **Memory (networkx):** ~2-4GB for full in-memory graph

For operations that don't need the full graph (most queries), we load only the relevant subgraph:
1. Query SQLite for seed nodes
2. BFS/DFS in SQLite to discover the relevant subgraph
3. Load only those nodes/edges into a networkx subgraph
4. Run the analysis on the subgraph

This reduces memory for typical queries to 10-100MB.

## 10. Error Handling

### 10.1 Error Categories

| Scenario | Handling | User Feedback |
|---|---|---|
| **Parse failure** (syntax error in source) | Log warning, skip file, continue analysis | Listed in `status --errors` output. Parse error count in `analyze` summary. |
| **Unresolvable reference** | Create edge with `confidence: "INFERRED"`, `resolved: false` | Not reported unless `--verbose`. Counted in `analyze` summary. |
| **Circular imports** | Detected and reported during Phase 2 | Warning in `analyze` summary. Not an error (common in Python). |
| **File too large** (>1MB source) | Parse normally | Warning: "Large file detected: path (X MB)" |
| **Corrupted `.code-atlas/`** | `code-atlas analyze --rebuild` to wipe and rebuild | Error message with recovery instructions. |
| **Missing tree-sitter grammar** | Clear error at `analyze` time | "Missing grammar for Java. Install: pip install tree-sitter-java" |
| **Permission denied** (file unreadable) | Skip file, continue | Warning in summary. |
| **Encoding error** (non-UTF-8 file) | Skip file, continue | Warning in summary. |
| **Out of memory** | N/A for most queries (subgraph loading). Full graph load may fail at extreme scale. | Suggest using `--scope` or reducing analysis scope. |

### 10.2 Graceful Degradation

The tool should always produce useful results even with partial data:
- If 10% of files fail to parse, the graph is still 90% complete and most analyses work correctly.
- If a reference can't be resolved, the edge is still created (with `INFERRED` confidence) so analyses can still traverse it.
- Phase 2 resolution failures don't block Phase 1 results.

## 11. Testing Strategy

### 11.1 Test Layers

| Layer | Type | What's Tested | Approach |
|---|---|---|---|
| **Core models** | Unit | `GraphNode`, `GraphEdge`, `ParsedFile` dataclasses | Assert construction, serialization, hashing |
| **Graph model** | Unit | `CodeGraph` add/remove/query operations | Test with small hand-built graphs |
| **Persistence** | Unit | SQLite read/write, incremental update logic | Test with temporary databases, assert round-trip |

| **Java adapter** | Unit | Node/edge extraction from Java source | Small fixture files with specific constructs. Assert exact nodes/edges. |
| **Reference resolution** | Integration | Cross-file reference resolution | Multi-file fixture projects. Assert resolved edges. |

| **End-to-end** | E2E | Full pipeline on small real projects | Small Flask app + Spring Boot app fixtures. Run `init` → `analyze` → queries. |

## 14. Future Extensions (Out of Scope for v1)

These are explicitly not in v1 but designed to be easy to add:

1. **Circular dependency detection** — Trivial to add: `networkx.find_cycle()` on the package graph.
2. **Hotspot analysis** — Compute betweenness centrality and degree centrality on the graph.
3. **MCP Server mode** — Wrap the analysis engine in an MCP server for AI assistant integration.
4. **File watcher mode** — Use `watchdog` to auto-update graph on file changes.
5. **Additional languages** — TypeScript, Go, Rust, Kotlin, C# via new adapters.
6. **Git integration** — Blame analysis, change frequency correlation with hotspots.
7. **Build manifest parsing** — Parse `pom.xml`, `requirements.txt`, `pyproject.toml` for external dependency tracking.
8. **Diff-based blast radius** — Accept a git diff and compute blast radius for the entire changeset.
9. **IDE plugins** — VS Code / IntelliJ extensions querying the Code Atlas graph.
10. **LLM-powered queries** — Natural language questions translated to graph traversals.

## 15. Glossary

| Term | Definition |
|---|---|
| **FQN** | Fully Qualified Name — unique identifier for a code entity (e.g., `myapp.services.UserService.create_user`) |
| **CST** | Concrete Syntax Tree — the parse tree produced by tree-sitter |
| **AST** | Abstract Syntax Tree — simplified version of CST (tree-sitter produces CSTs, but the term AST is used loosely) |
| **Adapter** | A language-specific module implementing the `LanguageAnalyzer` protocol |
| **Phase 1** | Per-file parsing and local extraction (parallelizable) |
| **Phase 2** | Cross-file reference resolution (sequential, requires full graph) |
| **Blast radius** | The set of entities transitively affected by a change to a given entity |
| **Entry point** | A function that serves as a program entry (main, HTTP handler, test, CLI command) |
| **Afferent coupling (Ca)** | Number of packages that depend on a given package |
| **Efferent coupling (Ce)** | Number of packages a given package depends on |
| **Instability (I)** | Ce / (Ca + Ce). 0 = maximally stable, 1 = maximally unstable |
| **Leiden algorithm** | Community detection algorithm for finding densely connected groups in a graph |
